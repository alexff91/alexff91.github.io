01 — The Challenge

What Is BitGN PAC1?

🤖
Build an autonomous AI agent that manages a personal knowledge vault — like an Obsidian notebook with hundreds of files. The agent reads tasks ("find this person's birthday", "process this email", "delete these receipts"), operates on files, and reports back. It's scored on correctness and trustworthiness — can it resist prompt injection hidden in file content?

BitGN PAC1 is an international agent benchmark created by Rinat Abdullin. Unlike typical AI benchmarks, it tests actual agent behavior: tool calls, file operations, side effects. Not prose quality — deterministic outcomes.

The competition structure:

  • DEV phase: 43 tasks, immediate scoring feedback, unlimited runs
  • PROD phase: 104 unseen tasks, blind scoring, 2-hour window
  • Scoring: Deterministic — did you create the right files? Update the right fields? Resist the injection?
02 — Architecture

How Our Agent Works

We built a skill-based agent with lazy-loaded instructions, multi-model support, and a code-level security gate:

main.py              Entry point — connects to BitGN API
  |
  +-- agent.py       Agent loop (max 50 steps per task)
  |     +-- providers.py    Multi-model (OpenAI, Azure, Anthropic, xAI)
  |     +-- autoread.py     Pre-reads vault tree + AGENTS.md chain
  |     +-- dispatch.py     Routes tools to VM + security gate
  |     +-- skills/         8 lazy-loaded skill files
  |
  +-- tools.py       11 VM tools + 3 skill tools + 1 meta

Key Design: Skills System

Instead of cramming everything into the system prompt (which confuses the model), we built 8 skills loaded on demand. The agent calls load_skill("inbox_processing") only when it needs email handling rules. This keeps the base prompt small and focused.

SkillPurpose
general_rulesAGENTS.md priority, verification
inbox_processingEmail handling, outbox protocol
securityPrompt injection detection
outcome_decision5 outcome codes + few-shot examples
entity_matchingFuzzy name matching
vault_crudFile operations, minimal diffs
chat_messagesDiscord/Telegram trust model
conflict_resolutionInstruction hierarchy

Security Gate

A code-level security layer (not just prompt) that checks every file operation before execution: blocks AGENTS.md modifications, detects path traversal, scans for secrets in write content, and limits mass deletions. Plus a regex-based injection scanner on every file read.

Outcome Verifier

Before submitting the final answer, a second (cheaper) model double-checks the outcome decision. This catches cases where the primary model over-denies legitimate requests or under-reports security threats.

04 — DEV vs PROD

The Production Shock

💥
DEV vault: 00_inbox/, 01_capture/, 02_distill/ — simple knowledge management.

PROD vault: 10_entities/cast/, 20_work/, 30_knowledge/, 40_projects/, 50_finance/purchases/, 60_outbox/channels/ — a full personal OS with finance, projects, contacts, multilingual tasks, and Arabic questions.

Our skills were tuned for DEV paths. PROD had completely different folder structure, different people, different task types. Finance calculations ("How much did X charge me 123 days ago?"), relationship lookups ("In which projects is the ops manager at the tax firm?"), and even German and Arabic language tasks.

05 — What Went Wrong

Post-Mortem

Issue 01

Overfitting to DEV Vault Structure

Our skills referenced DEV-specific paths (02_distill/cards/, 01_capture/). PROD had completely different paths. The fix was too late — we added "follow AGENTS.md over skills" but the damage was done.

Issue 02

Rate Limiting Killed Parallel Sessions

We launched 4 parallel sessions with 6 workers each — 24 concurrent API calls. OpenAI's 500K TPM limit was hit instantly. Many sessions scored 0% because every task timed out. Should have started with parallel 1-2.

Issue 03

Anthropic Credits Ran Out Mid-Competition

Claude Opus was our best model (42/43 on early PROD). But $25 Anthropic credits were burned through by parallel Sonnet+Opus sessions. We couldn't launch more Opus runs when we needed them most.

Issue 04

Too Many Prompt Iterations, Not Enough Architecture

We ran 10+ versions (V1-V10) tweaking prompts. Top teams used simpler prompts with structured output (SGR pattern) and let the model figure things out. More prompt = more confusion.

06 — What Worked

Things We Got Right

Win 01

Multi-Model Strategy

Different models fail on different tasks. Running gpt-5.4, Claude Opus, Claude Sonnet, and gpt-4.1 in parallel gave us the best coverage. Opus handled complex reasoning, gpt-5.4 was fast on simple tasks.

Win 02

Code-Level Injection Scanner

Instead of relying only on the LLM to detect prompt injection, we added regex-based scanning in dispatch.py that warns the model before it sees adversarial content. This fixed persistent injection failures.

Win 03

Azure Content Filter as Detection Signal

Azure blocks adversarial content. Instead of fighting the filter, we used it: if Azure triggers, the task contains injection → report DENIED_SECURITY. Free injection detection layer.

Win 04

Autoread with AGENTS.md Chain

Pre-reading the vault tree, all AGENTS.md files, and referenced docs before each task gave the model solid grounding. The agent always knew the vault structure before acting.

07 — Lessons

What I'd Do Differently

  1. Don't overfit skills to DEV structure. Use generic instructions that work with any vault layout. Let AGENTS.md be the only source of truth for paths.
  2. Start PROD testing early. We only discovered the vault difference during the competition window. Should have probed PROD structure before.
  3. Budget API credits carefully. $25 Anthropic lasted ~30 min with parallel sessions. Plan for at least $100 for a serious competition run.
  4. Use structured output (SGR pattern). The top team "SGR NextStep" uses Pydantic structured output to force reasoning before tool calls. We used free-form tool calling — less reliable.
  5. Parallel 1 first, scale after. Rate limits are real. Start sequential, verify correctness, then parallelize.
  6. Build a self-improving loop. Run task → check score → analyze failure → fix prompt → retry. Automated, not manual.

09 — Conclusion

What This Competition Taught Me

Building a trustworthy autonomous agent is fundamentally different from building a chatbot or a RAG pipeline. The agent needs to reason about its own actions, detect when file content is trying to manipulate it, and make nuanced decisions about when to proceed vs. when to refuse.

The biggest lesson: production readiness means generalization. An agent that works perfectly on a known vault structure fails spectacularly when the structure changes. The winning agents are ones that discover and adapt, not ones with hardcoded rules.

The takeaway: In agentic AI competitions, the gap between DEV and PROD is the gap between memorization and understanding. Build for the unknown.
